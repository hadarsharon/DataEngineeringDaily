Introduction:

My website is called Data Engineering Daily, and as the name suggests - it is aimed at Data Engineering enthusiasts with any of the following ideas in mind:

  • Following news and releases related to Data Engineering and/or Open Source technologies
  • Learning new things about Data Engineering, or reinforcing existing knowledge & skills
  • Discussing Data Engineering-related topics, questions, ideas etc. in the Website's forum
  • Subscribing to the website for a daily newsletter/digest, so that one is always up to date

The idea came to me while contemplating the site's theme, as I am a Data Engineer by trade.
I figured that if I do Data Engineering every day, it is probably the topic I'd be most comfortable with.
It was also very important for me to add a lot of relevant original content made entirely by me, and I can say that it was an interesting experience for me coming up with such.

I chose to implement a homepage-heavy website, where most of the content and interesting stuff happen in the home page.
That is because I believe that a website's home page should be eye-catching and engaging, so that the user would like to spend more time in the website and explore it.
If the home page is bland or off-putting - the user might as well assume the rest of the pages are just as bland and won't take the time to explore the website, instead opting for more eye-catching ones.
I also made sure that a lot of the elements are clickable or interactive - links, videos, images etc.

Despite that, I think the website is very clearly sectioned and there is quite a lot of space for all the elements to fit together.
I made it with the intention for it not to be cramped and overwhelming, but open and flowing.


State 3 things that have inspired you when creating your website:

When creating my website, I was inspired by the following:

  1. Local news sites that I visit on a daily basis - I thought that the fact I come back to these specific sites means there is something about their design & layout that clicks with me.
  2. Code/Frameworks/Modules/Libraries documentation sites - Funnily enough, I spend a lot of time reading documentation, and a lot of the style and layout is similar in these.
    I thought that I surely I am not the only one who is used to that style, and that it might fit the theme of the website perfectly.
  3. The Midterm Group Project - Although the Midterm Group project was not nearly as detailed as this one,
    I think that I learned a from it and took away two parallel notions from it:
    • What did we do well - and how I can apply these things in this project as well
    • What we did NOT do well - and how I can learn from our mistakes and avoid them in this project

State 3 ways in which your site is accessible:

My website is accessible in the following ways:

  1. Almost every img element has an appropriate 'alt' attribute, so that sight-impaired visitors would not be
    left wondering what these elements represent and what their goal in the page is
  2. Although not directly implemented by me but rather vicariously - the embedded YouTube videos contain a built-in "Subtitles" feature, as well as automatic Subtitles generation (at one's discretion) provided by YouTube.
    This would allow hearing-impaired visitors to watch the embedded videos with ease.
  3. The text and visual aspects of my website are styled in a very bold and clear theme - large fonts, very colorful etc.
    This would make it easier for older audience to interact with the site, as it is quite predictable and straightforward, without many hidden or unxpected quirks.

State 3 ways in which you have considered usability in your site:

I considered usability in the following contexts:

  1. Interactive elements are intuitive and "inviting" - i.e. there are buttons involved in almost all clickable elements, arrows for slideshows, play button for videos etc.
  2. Dynamic use of font size and horizontal rules - Every section has an appropriate header that separates the content from the section within which its contained,
    Horizontal rules separate the sections and create a clear visual border between different areas of the page.
  3. Using Borders and Modals - Border clearly define where the content starts and where it ends, while Modals pop up and display ONLY the relevant content, ONLY when asked to (e.g. at button click).

State 3 things you had to learn/find out by yourself when creating your site, and describe how you did that

  1. Element Animations - for my website to be interactive and engaging, I had to find out how to implement animations and transitions (e.g. slideshow).
    I went through some online guides on the subject and also looked at a lot of example code until I understood what I like more and what I'd rather not use.
    It was also important to me to keep things relatively simple and implement everything myself, so as to not use other people's code, especially if I don't understand it.
  2. Handlebars templating - Since my website has a lot of original content, and I do not have access to a Web Server, I had to make use of a client-side templating engine such as Handlebars.
    although we were introduced to Handlebars during the semester, I did not remember all of its nuts and bolts and had to go back to the lectures.
    Of course, I also went over the API reference and examples provided in the Handlebars website.
    This was crucial for me, as hard-coding all of my original content without the help of Handlebars would have made the HTML code very verbose and bloated.
  3. Bootstrap and jQuery - It is my opinion that I am not good with CSS, therefore any framework that could help me structure my webpages as intended is a blessing.
    I chose Bootstrap because it is intuitive and usable, and it helped me create the structure I needed without writing a lot of boilerplate CSS.
    It also helped me implement responsiveness in my website.
    As for jQuery - it helped me cut down on the amount of JavaScript I had to write, and had a lot of useful methods that helped me make my site more entertaining and interactive with less code.

What aspect(s) of your work do you think worked well and why?

I think that the website's HTML is relatively well-structured and that its JavaScript code is concise and efficient.
The website loads rather quickly and feels natural and fast to navigate around, which is important for every site.
Given that I wrote most of the content myself, I feel that the website feels authentic and relevant, without almost any placeholders or lorem ipsum-style content.
I especially like the "Quizzes" section, as it was fun for me to come up with the questions and I think it is the most enjoyable part of the site.


What aspect(s) of your work could be improved, and how might you do things differently another time?

As stated earlier, I definitely feel that, out of the 3 languages used in this project (HTML, CSS, JS), I am by far the least comfortable with CSS.
I have a feeling that this might be reflected in the website, both in terms of responsiveness and perhaps also in the quality of the CSS code.
Of course, I did my best, however if there's any specific area I'd like to improve on is CSS, responsiveness and styling.


What resources were used when creating your work?

I used the following resources:

  • Bootstrap 4 (Used for Grid and Structure)
  • jQuery (Used for JavaScript functions and event handlers)
  • Handlebars.js (Used as a templating engine for embedding original content in HTML)
  • Font Awesome (Used for eye-catching icons such as Social Media icons at the footer)
  • W3Schools (Used mostly for reference and tutorials, but no code copied or re-used except for the "Login" popup modal, which was used as a template and modified by me.
  • All non-original content is open source and taken from either free Google results (Shutterstock and similar free website) or directly from the Open Source project's respective website, for example the release notes for Apache Airflow 2.0
